## 1. Dataset at a glance

We replicate the dataset released by **Kocabey et al. (2017, ICWSM, *Face-to-BMI*)**: weight-change before/after face crops with self-reported BMI and gender labels.

| Stage | Count | Retention vs raw |
|---|---:|---:|
| Raw label rows (`data/raw/data.csv`) | 4,206 | 100.00% |
| Image files on disk (`data/raw/Images/*.bmp`) | 3,962 | 94.20% |
| Clean labels (rows with file present) | 3,962 | 94.20% |
| Canonical split rows (`split_ids.csv`) | 3,962 | 94.20% |
| Standardized 224×224 OK | 3,962 | 94.20% |
| MTCNN-aligned OK (v3 pilot) | 3,962 | 94.20% |

**Canonical split (frozen contract):** `split_ids.csv` provides 3,210 train / 752 test rows. The split is **pair-aware**: both before/after images of the same `person_id` are placed in the same split, so the leakage check `persons_in_multiple_splits == 0` holds. Do not regenerate.

## 3. Preprocessing strategies

We maintain **two parallel batch preprocessing pipelines** (v1 and v2/v3) and **one lightweight inference pipeline** for the Web API. They produce different on-disk artifacts and serve different consumers.

### 3.1 v1 — Simple standardization → `data/processed/faces_standardized/`

Implemented in [`notebooks/data/01_audit_and_standardize.ipynb`](notebooks/data/01_audit_and_standardize.ipynb).

**Steps**
1. Audit every raw image (existence, dimensions, mode, aspect ratio).
2. **Pad to square** with **black bars** on the shorter sides (no stretching, no rotation).
3. **Resize the square** to **224 × 224** RGB, written as `.bmp`.

**Rationale.** Faces in the dataset are already framed reasonably tightly. Running a heavy detect-and-align pass on every image risked introducing more artifacts than signal. Pad-then-resize guarantees aspect-ratio preservation with the minimum amount of black padding required to reach a 224 × 224 input.

**Output.** 3,962 standardized RGB BMPs. This is the **frozen training-time input distribution** for Beste's regressors, and the contract any new code must conform to before feeding the SVRs.

### 3.2 v2 / v3 — MTCNN landmark alignment → `data/processed/faces_aligned/` and `faces_aligned_v3_pilot/`

Implemented in [`notebooks/data/01b_audit_and_align_mtcnn.ipynb`](notebooks/data/01b_audit_and_align_mtcnn.ipynb).

**Why v2 exists.** v1 produced a small horizontal stretch on a subset of images where the original was wider than it was tall. The fix: keep the **final resize square-to-square** by cropping a square region around the detected face, padding only when the square would run off the source edge.

**Steps**
1. Run **MTCNN** on the original image to locate the face and the five landmark points (eyes, nose, mouth corners).
2. **Rotate** so the inter-ocular line is horizontal (eye-leveling alignment).
3. Crop a **square region centered on the face**, sized to keep the standard amount of context.
4. **Pad the source only as an edge-case safety net** when the square region runs off the image boundary. Most images have well-centered faces and are not padded.
5. Resize the square crop to 224 × 224 RGB.

**Outputs**
- `faces_aligned/` — full MTCNN-aligned set.
- `faces_aligned_v3_pilot/` — pilot run of the v3 detection logic with the per-image status, rotation angle, padding fraction, and degenerate-eye-distance flag in `data/processed/detection_log_v3_pilot.csv`.

**Status.** v1 (`faces_standardized/`) remains the contract-bound model input; v2/v3 alignment is an alternative input branch under evaluation by the feature-extraction owner (Ryan, Role 3).

### 3.3 Inference-time preprocessing → [`src/inference_preprocess.py`](src/inference_preprocess.py)

The Web API needs single-image, low-latency preprocessing. `pipeline_v1.py` (Wade) is correct for batch evaluation but heavy for a request hot path (facenet-pytorch / TF cold start ≈ 1.5 s, ~200 MB RAM). `inference_preprocess.py` stays in PIL + numpy + cv2 only and produces an array byte-for-byte compatible with v1 standardization.

**Steps**
1. **Decode** any of: filesystem path, bytes, base64 / data-URL string, NumPy array, PIL image, file-like object.
2. Validate size (`min_side=32`, `max_side=8192`, `max_bytes=10 MiB`) and apply **EXIF rotation** (so portrait phone uploads land upright).
3. **Optional face crop** with an **OpenCV Haar cascade** (frontal-face), expanded by a 20% margin to keep the same loose framing as the training inputs. Falls back to no-crop if no face is detected — the SVRs are tolerant.
4. **Pad-to-square (black bars) + resize to 224 × 224 RGB** — a direct port of the v1 pad-resize procedure.
5. Return a `uint8` ndarray of shape `(224, 224, 3)`, optionally with diagnostics (original size, detected box, per-stage timings).

**Why Haar instead of MTCNN at inference?** Haar is already a transitive dependency of `cv2`, runs in tens of milliseconds on CPU, and adds zero new heavy imports. MTCNN would dominate first-request latency without a measurable downstream r improvement (the SVRs were trained on the v1 pipeline, not the v2/v3 alignment).

## 4. Frozen contracts (for the report's reproducibility section)

| Contract | Value | Owner / consumer |
|---|---|---|
| Standardization target | 224 × 224 RGB, aspect-preserving pad-to-square with **black bars** | Zihao → Beste |
| DeepFace call | `model_name='VGG-Face' \| 'Facenet512'`, `enforce_detection=False`, `detector_backend='skip'` | Beste |
| Embedding dimensions | VGG-Face: 4,096-D · Facenet512: 512-D | Beste → Ryan |
| Train/test split | `split_ids.csv` (3,210 / 752, pair-aware, `persons_in_multiple_splits == 0`) | All |
| Random seed | `random_state=42` everywhere (numpy / sklearn / torch) | All |

Inference replicates training exactly. Any deviation here surfaces at integration time as a silent drop in Pearson *r* relative to Beste's reported numbers (VGG *r* = 0.6261, ensemble *r* = 0.6469 vs the paper's 0.65 baseline).
